4 <P>The Hardware Abstraction Layer (HAL) is program code between a system's hardware and its software that provides a consistent interface
5 for applications that can run on several different hardware platforms. To take advantage of this capability, applications should access
6 hardware through the API provided by the HAL, rather than directly. Then, when you move to new hardware, you only need to
7 update the HAL. In some cases, due to extreme differences in hardware, the HAL API may also change slightly to accommodate the
8 new hardware. In these cases, the limited scope of the update makes moving the application easier with the HAL than without.</P>

12 <P>HAL function names have the following prefix conventions:</P>
13 
14  <UL>
15  <LI><b>halCommon:</b> API that is used by the EmberZNet stack and can also be called
16  from an application. This API must be implemented. Custom
17  applications can change the implementation of the API but
18  its functionality must remain the same.
19 
20 <LI><b>hal:</b> API that is used by sample applications. Custom
21  applications can remove this API or change its
22  implementation as they see fit.
23 
24 
25 <LI><b>halStack:</b> API used only by the EmberZNet stack. This API must be implemented
26  and should not be directly called from any application.
27  Custom applications can change the implementation of the
28  API, but its functionality must remain the same.
29 
30 <LI><b>halInternal:</b> API that is internal to the HAL. The EmberZNet stack
31  and applications must never call this API directly.
32  Custom applications can change this API as they see
33  fit. However, be careful not to impact the functionalty of
34  any halStack or halCommon APIs.
35  </UL>
