1 RAIL Multiprotocol
2 =====================
3 
4 # Overview {#rail_multiprotocol}
5 
6 As of version 2.1, the RAIL library supports dynamic multiprotocol. In dynamic
7 multiprotocol, an application can allocate multiple RAIL instances and configure
8 them independently. Some radio operations are then arbitrated by a radio
9 scheduler, which is internal to RAIL, to allow the different instances to coexist.
10 The scheduler attempts to allow as many of these operations to run as possible
11 by moving them around within the bounds allowed by the protocol.
12 
13 # Radio Scheduler {#rail_multiprotocol_radio_scheduler}
14 
15 The radio scheduler is critical for multiprotocol operation. Its job is to
16 arbitrate all radio operations, switch radio configurations, and ensure that
17 radio operations run at the time requested. Not every RAIL function is a radio
18 operation considered by the scheduler. Simple state update functions, such as \ref
19 RAIL_SetTxPower(), will directly change the hardware if active or update the
20 state cache if inactive to have that change take affect the next time this
21 protocol is loaded. The following RAIL functions are handled by the scheduler:
22 
23  - \ref RAIL_StartRx()
24  - \ref RAIL_ScheduleRx()
25  - \ref RAIL_StartTx()
26  - \ref RAIL_StartScheduledTx()
27  - \ref RAIL_StartCcaCsmaTx()
28  - \ref RAIL_StartCcaLbtTx()
29  - \ref RAIL_StartAverageRssi()
30  - \ref RAIL_StartTxStream()
31 
32 To arbitrate the radio operations, the scheduler uses
33 startTime, priority, slipTime, and transactionTime. startTime comes
34 implicitly from the RAIL API that started the transaction. For example, calling
35 \ref RAIL_StartTx() will use a start time of right now while calling \ref
36 RAIL_StartScheduledTx() takes an explicit start time as a parameter. The other
37 three parameters are passed to all RAIL functions that require the scheduler in
38 the \ref RAIL_SchedulerInfo_t structure. The scheduler attempts to run each task
39 at its desired startTime, but may also choose to run it up until startTime +
40 slipTime. If it cannot run the task within that window, it will trigger a \ref
41 RAIL_Config_t::eventsCallback with the \ref RAIL_EVENT_SCHEDULER_STATUS event
42 set once that window has passed.
43 
44 The scheduler is preemptive and will always ensure that a higher-priority
45 task runs over a lower-priority task, but it does not guarantee ordering of
46 these tasks. For example, it could choose to run a short lower-priority task
47 with a small slipTime before a higher-priority task with a long slipTime to reduce
48 dropped operations. Once an operation is started by the radio scheduler, it will
49 either run to completion and trigger a normal RAIL event callback to indicate it
50 is done or it will be aborted by a higher-priority task and ended with a \ref
51 RAIL_EVENT_SCHEDULER_STATUS event. Ultimately, the application is responsible
52 for terminating most operations with a call to \ref RAIL_YieldRadio(). This call
53 cleans up internal state and allows the scheduler to switch to a lower priority
54 operation.
55 
56 \note For the \ref RAIL_SchedulerInfo_t::priority parameter, 0 is the highest
57 priority while 255 is the lowest.
58 
59 ## Scheduler Operations
60 
61 The APIs mentioned above can be classified as finite
62 operations, infinite operations, and debug operations.
63 
64 ### Finite Operations
65 
66 Finite operations include the following APIs:
67 
68  - \ref RAIL_ScheduleRx()
69  - \ref RAIL_StartTx()
70  - \ref RAIL_StartScheduledTx()
71  - \ref RAIL_StartCcaCsmaTx()
72  - \ref RAIL_StartCcaLbtTx()
73  - \ref RAIL_StartAverageRssi()
74 
75 Finite operations make up the majority of the radio scheduler calls. A finite
76 operation has a defined start time and either a defined or estimated end
77 time. Each RAIL handle may only have one of these types of operations queued up
78 at any time. The upper level application code is responsible for waiting until
79 the previous operation has completed before starting a new one like in the
80 single protocol RAIL library. If a new task is issued while a previous task is
81 in progress, or if a higher-priority task interrupts an operation before the user
82 has yielded, it will trigger the \ref RAIL_EVENT_SCHEDULER_STATUS event with an
83 appropriate \ref RAIL_SchedulerStatus_t set and this operation will not be
84 scheduled again.
85 
86 Once a finite operation has begun, the scheduler will use the same priority until
87 a new finite event is provided or a yield occurs. The scheduler will ensure that
88 unnecessary switches do not occur if this next event is in the near future at
89 the same or a higher priority. For more information see \ref
90 rail_radio_scheduler_yield.
91 
92 ### Infinite Operations
93 
94 Infinite operations include the following APIs:
95 
96  - \ref RAIL_StartRx()
97 
98 An infinite or background operation is treated by the scheduler as a
99 radio state change. It still has a priority associated with it and will be
100 started whenever it can be, but it will never fail. Another difference is that
101 if this event is interrupted by a higher-priority task, it will be restarted
102 whenever that task finishes where a finite task would be aborted.
103 
104 Each RAIL handle may have only one infinite task but it may have both an
105 infinite and finite or debug task at the same time. The one interesting aspect
106 of this is that, if a finite task starts for this RAIL handle, the state will be
107 updated to reflect what the infinite task has requested for that finite task's
108 time period even if the infinite task is lower priority than another task in the
109 system. For example, if you have one handle with a RAIL_StartRx() of priority 5
110 and another handle with a RAIL_StartRx() priority of 6 and a RAIL_StartTx()
111 priority of 3, the scheduler will choose the second handle's finite transmit
112 task. While running this transmit task though, the receiver will be set to on so
113 that, as soon as this transmit completes, the radio will transition from transmit
114 into receive. It will remain in receive until the finite operation yields at
115 which time the first handle's receive will take precedence.
116 
117 This implementation is intended to get the radio to stay in receive
118 for a very long time on one of the RAIL handles. It is generally only advisable
119 to do this on one radio configuration at a time and for this to be the lowest-priority
120 task. For well defined receive events, such as looking for an ACK scheduled
121 receive, windows or state transitions will generally work better.
122 
123 \note A known bug exists with turning on an infinite task during a finite
124 task. For now, you must always configure the infinite task first to ensure it
125 takes effect.
126 \note If using state transitions that can lead from receive to idle, the user
127 must tell the scheduler that this happened with a call to \ref RAIL_Idle().
128 This restriction will hopefully be removed in future versions.
129 
130 ### Debug operations
131 
132 Debug operations include the following APIs:
133 
134  - \ref RAIL_StartTxStream()
135 
136 Debug operations are somewhat special because they don't allow the user to pass
137 a \ref RAIL_SchedulerInfo_t structure to them. Instead, they implicitly provide
138 a start time of now and use the highest possible priority. This means that they
139 will have the highest priority and interrupt most other protocol operations to
140 switch into the test mode. Nothing can preempt them once started because the
141 scheduler does not allow tasks of equal priority to preempt. These operations
142 can be stooped by calling \ref RAIL_StopTxStream(), \ref RAIL_YieldRadio(), or
143 \ref RAIL_Idle().
144 
145 The modal nature of these debug operations is intentional to prevent multiple
146 tone or stream operations from happening concurrently. It is expected that the
147 user will manage these test modes from a higher level to prevent the different
148 protocols from using tone or stream at the same time.
149 
150 ## Yielding the radio {#rail_radio_scheduler_yield}
151 
152 Yielding is used by finite radio scheduler operations. Once an operation is
153 started, that operation will run until explicitly yielded by the upper layer or
154 interrupted by a higher-priority task. A yield is forced by a call to \ref
155 RAIL_YieldRadio() or \ref RAIL_Idle(). It is important that the application
156 yield the radio as soon as it's done to allow lower-priority tasks to run.
157 
158 Yielding is not handled by RAIL since terminating an operation can be
159 application-specific. While RAIL may know that an individual transmit or receive
160 operation is completed, there are sometimes other reasons to prevent a switch or
161 to run a follow on operation. For example, if you are using ACKing you may want
162 to wait for the \ref RAIL_EVENT_TXACK_PACKET_SENT after a receive packet instead
163 of yielding right after the receive. This also allows you to chain together many
164 protocol-specific operations without the scheduler constantly context switching.
165 If you do schedule a follow on operation that is far in the future, the scheduler
166 will also be smart enough to treat this as a yield and slot in other lower-priority
167 operations.
168 
169 This does mean that you must carefully look through the \ref Events in RAIL and
170 subscribe to any that can impact your state machine. Specifically, things like
171 transmit or receive success conditions, transmit and receive error conditions,
172 and scheduler events (\ref RAIL_EVENT_SCHEDULER_STATUS, \ref
173 RAIL_EVENT_CONFIG_UNSCHEDULED, and \ref RAIL_EVENT_CONFIG_SCHEDULED) are
174 important to keep track of. These would likely be needed in any upper layer to
175 properly update the internal state machine as well so they're likely already in
176 the implementation.

178 # Building a Multiprotocol Application {#rail_multiprotocol_building}
179 
180 The RAIL API is the same whether the targeted application is using the
181 multiprotocol version or not. This was done intentionally to
182 make switching between the two versions simple. From a build perspective,
183 choose the desired RAIL library binary file to link
184 into the application. For multiprotocol applications, the file name is
185 librail_multiprotocol_CHIP_COMPILER_release.a while for other applications the
186 file name is librail_CHIP_COMPILER_release.a.
187 
188 Optionally, non multiprotocol applications can save memory by not allocating the
189 \ref RAIL_Config_t::scheduler state structure and passing NULL for all \ref
190 RAIL_SchedulerInfo_t pointers in APIs.
191 
192 # Example
193 
194 Below is a simple example showing how to initialize two RAIL configurations and
195 have one stay in infinite receive while the other one periodically transmits a
196 packet at a higher priority.
197 
198 ~~~~~~~~~~~~~~~{.c}
199 #include <stdint.h>
200 #include <stdbool.h>
201 #include "rail.h"
202 #include "rail_config.h" // Generated by radio calculator
203 
204 static RAIL_Handle_t gRailHandle1 = NULL;
205 static RAIL_Handle_t gRailHandle2 = NULL;
206 
207 // Each RAIL handle must have its own non-const configuration structure since state
208 // is stored in these variables.
209 
210 // Creates structure 1.
211 static RAILSched_Config_t railSchedState1;
212 static RAIL_Config_t railCfg1 = { // Must never be const
213  .eventsCallback = &radioEventHandler,
214  .protocol = NULL, // For BLE, pointer to a RAIL_BLE_State_t
215  .scheduler = &railSchedState1,
216 };
217 // Creates structure 2.
218 static RAILSched_Config_t railSchedState2;
219 static RAIL_Config_t railCfg2 = { // Must never be const
220  .generic = &radioEventHandler,
221  .protocol = NULL, // For BLE, pointer to a RAIL_BLE_State_t
222  .scheduler = &railSchedState2,
223 };
224 
225 // Boolean to track when a packet has been sent.
226 static bool packetSendComplete = true;
227 
228 // Creates transmit FIFOs for each PHY.
229 #define TX_FIFO_SIZE 128
230 static uint8_t txFifo1[TX_FIFO_SIZE];
231 static uint8_t txFifo2[TX_FIFO_SIZE];
232 
233 static void radioConfigChangeHandler(RAIL_Handle_t railHandle,
234  const RAIL_ChannelConfigEntry_t *entry)
235 {
236  bool isSubgig = (entry->baseFrequency < 1000000000UL);
237 
238  // ... handle radio configuration change, e.g., select the desired PA possibly
239  // using isSubgig to handle multiple configurations.
240  RAIL_ConfigTxPower(railHandle, &railTxPowerConfig);
241 
242  // Reapply the Tx power after changing the PA above.
243  RAIL_SetTxPowerDbm(railHandle, txPower);
244 }
245 
246 static void radioEventHandler(RAIL_Handle_t railHandle,
247  RAIL_Events_t events)
248 {
249  // Note that two different callbacks above could be used,
250  // but this example only uses one which is split based on the handle.
251  if (railHandle == gRailHandle1) {
252  // Handle events for protocol 1.
253  // NOTE: Since in infinite receive, the radio does not have to be yielded
254  // here as below. If this were changed, the radio would have to be yielded here.
255  } else if(railHandle == gRailHandle2) {
256  // Handle any packet completion event (success or failure) and set us up
257  // to send another packet.
258  if (events & (RAIL_EVENT_TX_PACKET_SENT
259  | RAIL_EVENT_TX_ABORTED
260  | RAIL_EVENT_TX_UNDERFLOW
261  | RAIL_EVENT_SCHEDULER_STATUS)) {
262  packetSendComplete = true;
263  RAIL_YieldRadio(railHandle);
264  }
265  }
266 }
267 
268 // Initializes the two PHY configurations for the radio.
269 void radioInitialize(void)
270 {
271  // Creates each RAIL handle with their own configuration structures.
272  gRailHandle1 = RAIL_Init(&railCfg1, NULL);
273  gRailHandle2 = RAIL_Init(&railCfg2, NULL);
274 
275  // Sets up transmit FIFOs.
276  RAIL_SetTxFifo(gRailHandle1, txFifo1, 0, TX_FIFO_SIZE);
277  RAIL_SetTxFifo(gRailHandle2, txFifo2, 0, TX_FIFO_SIZE);
278 
279  // Configures radio according to the generated radio settings.
280  RAIL_ConfigChannels(gRailHandle1, channelConfigs[0], &radioConfigChangedHandler);
281  RAIL_ConfigChannels(gRailHandle2, channelConfigs[1], &radioConfigChangedHandler);
282 
283  // Configures the most useful callbacks plus catch a few errors.
284  RAIL_ConfigEvents(gRailHandle1,
285  RAIL_EVENTS_ALL,
286  RAIL_EVENT_TX_PACKET_SENT
287  | RAIL_EVENT_TX_ABORTED
288  | RAIL_EVENT_TX_UNDERFLOW
289  | RAIL_EVENT_SCHEDULER_STATUS
290  | RAIL_EVENT_RX_PACKET_RECEIVED
291  | RAIL_EVENT_RX_FRAME_ERROR // invalid CRC
292  | RAIL_EVENT_RX_ADDRESS_FILTERED);
293  RAIL_ConfigEvents(gRailHandle2,
294  RAIL_EVENTS_ALL,
295  RAIL_EVENT_TX_PACKET_SENT
296  | RAIL_EVENT_TX_ABORTED
297  | RAIL_EVENT_TX_UNDERFLOW
298  | RAIL_EVENT_SCHEDULER_STATUS
299  | RAIL_EVENT_RX_PACKET_RECEIVED
300  | RAIL_EVENT_RX_FRAME_ERROR // invalid CRC
301  | RAIL_EVENT_RX_ADDRESS_FILTERED);
302 }
303 
304 int main(void)
305 {
306  RAIL_SchedulerInfo_t schedulerInfo;
307  uint32_t sendTime;
308 
309  // Initializes the radio and creates the two RAIL handles.
310  radioInitialize();
311 
312  // Only set priority because transactionTime is meaningless for infinite
313  // operations and slipTime has a reasonable default for relative operations.
314  schedulerInfo = (RAIL_SchedulerInfo_t){ .priority = 200 };
315  RAIL_StartRx(gRailHandle1, 0, &schedulerInfo);
316 
317  // Starts the first send 2 seconds after getting set up.
318  sendTime = RAIL_GetTime() + 2000000;
319 
320  // Issues a transmit periodically on the second RAIL handle.
321  while (true) {
322  if (packetSendComplete) {
323  RAIL_Status_t res;
324  RAIL_ScheduleTxConfig_t scheduledTxConfig = { .when = sendTime,
325  .mode = RAIL_TIME_ABSOLUTE };
326  uint8_t packetData[] = { 0, 1, 2, 3, 4, 5, 6, 7, 7, 8, 10 };
327 
328  // This assumes the Tx time is around 10 ms but should be tweaked based on
329  // the specific PHY configuration.
330  schedulerInfo = (RAIL_SchedulerInfo_t){ .priority = 100,
331  .slipTime = 50000,
332  .transactionTime = 10000 };
333 
334  // Loads the transmit buffer with something to send.
335  RAIL_WriteTxFifo(gRailHandle2, packetData, sizeof(packetData), true);
336 
337  // Transmits this packet at the specified time or up to 50 ms late.
338  res = RAIL_StartScheduledTx(gRailHandle2,
339  0,
340  RAIL_TX_OPTIONS_DEFAULT,
341  &scheduledTxConfig,
342  &schedulerInfo);
343  if (res == RAIL_STATUS_NO_ERROR) {
344  packetSendComplete = false;
345  sendTime += 2000000; // Add 2 seconds to the previous time.
346  } else {
347  // In the current configuration this should never happen.
348  assert(false);
349  }
350  }
351  }
352 
353  return 0;
354 }
