7 ### 2.5.0
8  * Added a channel hopping feature with \ref RAIL_ConfigRxChannelHopping
9  and \ref RAIL_EnableRxChannelHopping. This feature allows users to
10  specify a list of channels to scan over during receive. This feature
11  allows for much faster scanning with less delay (about 100us) than
12  simply repeatedly calling \ref RAIL_StartRx() with different channels.
13  * Included two default transmit power curves in the library:
14  RAIL_TxPowerCurvesDcdc and RAIL_TxPowerCurvesVbat. When the pa-conversions
15  plugin is used, one of these can be passed to the plugin, based on the way
16  the power amplifier is powered.
17  * Added the ability to specify a Link Quality Indicator (LQI) conversion
18  callback with \ref RAIL_ConvertLqi such that the hardware's LQI value can be
19  changed before application consumption of that value.
20  * Fixed an issue where \ref RAIL_StopTxStream() was taking an extended amount of
21  time to complete.
22  * Certain received packet details in \ref RAIL_RxPacketDetails_t, which
23  were formerly available only for successfully received packets, are now
24  also available for unsuccessfully received packets.
25  * \ref RAIL_DataMethod_t and related documentation have been updated to
26  reflect that many FIFO_MODE concepts can now be used in PACKET_MODE,
27  and the distinction between these modes has become nearly invisible for
28  transmit. Also eliminated the previous restriction that both receive and
29  transmit \ref RAIL_DataMethod_t must match in the \ref RAIL_DataConfig_t
30  passed to RAIL_ConfigData().
31  * RAIL_ConfigData() no longer resets the receive FIFO when switching to
32  \ref RAIL_DataMethod_t::FIFO_MODE.
33  * Added a new \ref RAIL_TxOptions_t option \ref
34  RAIL_TX_OPTION_CCA_PEAK_RSSI to allow the user to use the peak RSSI instead
35  of the average RSSI over the CCA period during CSMA and LBT transmits. This
36  is only supported on EFR32xG12 and newer silicon.
37  * Added a new \ref RAIL_TxOptions_t option \ref
38  RAIL_TX_OPTION_ALT_PREAMBLE_LEN to allow the user to transmit with a
39  runtime-specified preamble length. This alternate preamble length is set
40  with \ref RAIL_SetTxAltPreambleLength.
41  * Added \ref Z_Wave PHY configurations and functionality to the RAIL library.
42  * Fixed an issue that could cause register corruption in very rare conditions
43  due to a bus access race condition. The specific registers impacted were
44  limited to those related to the radio and the corruption was most likely
45  when running the coherent IEEE802.15.4 PHY on the EFR32xG12 and EFR32xG13
46  platforms. On other platforms and PHY configurations the issue is possible
47  though very unlikely and never observed (id: 317234).
48  * Resolved a multiprotocol scheduler issue that could cause some interrupted
49  events to not be reported to the caller. Since these events never run there
50  are no other related events either so state machines relying on RAIL events
51  could get stuck in invalid states (id: 341124).
52  * Enhanced RAIL_SetRxFifoThreshold() and RAIL_SetTxFifoThreshold() to now
53  support a threshold value of \ref RAIL_FIFO_THRESHOLD_DISABLED to
54  explicitly disable a threshold previously established.
55  * Fixed a documentation issue where a variety of RAIL APIs were incorrectly
56  shown as "Referenced by RAIL_CopyRxPacket()".

70 ### 2.4.0
71  * Added ability to set custom receive FIFO sizes via the RAIL_SetRxFifo()
72  function. The new RAILCb_SetupRxFifo() callback may be used to override
73  the default FIFO initialization to save RAM.
74  * Added alternate functions to get packet details, in a way that helps lower
75  code size: RAIL_GetRxPacketDetailsAlt() and RAIL_GetTxPacketDetailsAlt().
76  These come with functions to adjust the returned time stamp to different
77  places in the packet: RAIL_GetRxTimePreambleStart(),
78  RAIL_GetRxTimeSyncWordEnd(), RAIL_GetRxTimeFrameEnd(),
79  RAIL_GetTxTimePreambleStart(), RAIL_GetTxTimeSyncWordEnd(), and
80  RAIL_GetTxTimeFrameEnd().
81  * Broke up the IR calibration code to save code size for Zigbee and BLE where
82  the results are stored on chip.
83  * Added a new \ref RAIL_SetRssiOffset() command to offset all RSSIs used by
84  RAIL to match any hardware offsets on a given board.
85  * Added a new \ref RAIL_StopTx() function to stop only pending or current
86  transmits without impacting receive.
87  * For multiprotocol, added the ability to update a task's priority after
88  it started with RAIL_SetTaskPriority().
89  * Added the ability to pre-apply a radio configuration to hardware before a
90  TX or RX operation begins via RAIL_PrepareChannel().
91  * Added the ability to query RAIL for the channel configuration currently
92  configured for use via RAIL_GetChannel().
93  * Fixed an issue where relative tasks with a non-zero slip time could start a
94  few hundred microseconds earlier than they were supposed to.
95  * Fixed an issue that could cause antenna diversity to not function correctly.
96  * Fixed an issue where RAIL_Idle() with \ref RAIL_IDLE_FORCE_SHUTDOWN could
97  cause an erroneous \ref RAIL_EVENT_TX_ABORTED event to fire.

107 ### 2.3.0
108  * Added a RAIL_CopyRxPacket() helper for use after RAIL_GetRxPacketInfo().
109  * Added a fast, but inconsistent, transition times mode. This can be activated
110  by setting state timings to 0.
111  * Resolved a potential RX FIFO corruption that could occur when calling
112  RAIL_HoldRxPacket() or RAIL_PeekRxPacket().
113  * A multi-timer API was added that allows for multiple software timers all
114  driven off the one hardware timer. This must be explicitly enabled and will
115  use more code than the standard APIs, but can be done through \ref
116  RAIL_ConfigMultiTimer(). Once in use, all timer APIs must use the multi
117  timer versions to prevent conflicts with the hardware timer.
118  * Added diagnostic functions RAIL_Verify() and RAIL_ConfigVerification() to
119  provide a means of verifying internal radio memory contents.
120  * Added alternative calibration APIs to RAIL_Calibrate, for potential code size
121  savings: RAIL_CalibrateTemp, RAIL_CalibrateIr, and RAIL_ApplyIrCalibration.
122  * Fixed an issue where requesting a small fixed CSMA backoff could result in a
123  very large value from being selected instead.
124  * Made PHY configuration improvements for BLE and Zigbee on the efr32xg14
125  platform.

127 ### 2.2.0
128  * In FIFO mode on the EFR32 the FIFO could only be read after 8 bytes were
129  received. This restriction has now been lifted though the user must be
130  careful and understand that in certain cases they could be reading appended
131  information and not packet bytes.
132  * On the EFR32XG14 some BLE packets showed up with invalid timestamps. These
133  are now filtered and reported as aborted to the user so that the stack
134  does not time sync to an invalid timestamp and lose the connection.
135  * Fixed an issue where state transition times would not be recomputed when
136  changing the PA ramp time on the fly.
137  * Fixed a regression in 2.1.1 that caused Frame Type based length configurations
138  to lock up the receiver. There was an issue where bits in the byte that were
139  not supposed to be a part of the frame type were being included some times
140  and could cause problems in the receive processing logic.
141  * Prevented RAIL_StartRx() from resetting the receive FIFO. This was a legacy
142  feature that was not documented and, though it generally worked, would cause
143  problems to users processing deferred receive events or users that called
144  RAIL_StartRx() in an interrupt disabled context. If this behavior was
145  required it can be replicated by idling the radio and then calling
146  RAIL_ResetFifo() with rxFifo set to true before any calls to RAIL_StartRx().
147  * Fixed an issue in the RAIL_IEEE802154_SetAddresses() function that caused it
148  to never set any long address except for the first one and to set that one
149  to the value given in the final long address entry.
150  * Updated the documentation for the IEEE 802.15.4 and BLE protocol functions
151  to show properly usage with the new RAIL 2.0 API.
152  * Dynamic Multiprotocol fixes:
153  * Added support for each protocol having auto ACK enabled by allocating
154  space for independent ACK buffers.
155  * Ensured that the sync word is properly output in BLE Long Range no matter
156  when the packet data is loaded into the transmit FIFO.
157  * Prevented the sync word in packet trace feature from being turned on in
158  Zigbee when it's enabled by BLE.

164 ### 2.1.1
165  * \ref RAIL_PauseTxAutoAck() was internally pausing the Rx AutoAck logic. This
166  has now been corrected.
167  * EFR32XG12 and EFR32XG13 devices were detecting a high number of false 15.4
168  sync words. Even though these packets were eventually filtered they would
169  show up on PTI. In 2.1.1 there should be significantly fewer of these
170  packets as some of this was caused by a bug in a configuration algorithm.
171  * Channel power limits for BLE PHYs were not always being respected. This
172  could have caused us to use a power which might result in certification
173  issues.
174  * The \ref RAIL_StateBuffer_t type was incorrectly sized resulting in a waste
175  of RAM for RAIL applications. This has now been trimmed down to the actual
176  size required.

178 ### 2.1.0
179  * Packet receive APIs were totally revamped to allow faster and out-of-order
180  deferred processing of received packets and their data outside of
181  interrupt/callback context. Gone are RAIL_GetRxPacketHandle(),
182  RAIL_ReadRxFifoAppendedInfo(), and the memory management RAILCb_ callbacks,
183  replaced with RAIL_GetRxPacketInfo() that provides pointers directly to
184  the received data held in RAIL's internal receive FIFO buffer allowing for
185  zero-copy operation. See other RAIL 2.x porting documentation and
186  new routines RAIL_GetRxPacketInfo(), RAIL_GetRxPacketDetails(),
187  RAIL_HoldRxPacket(), and RAIL_ReleaseRxPacket(), along with updated
188  RAIL_PeekRxPacket() and RAIL_ReadRxFifo() behavior.
189  * On the transmit side, zero-copy is also now supported by allowing
190  applications to specify their own buffers for the transmit FIFO,
191  eliminating RAIL's former internal fixed-size 512-byte one. See
192  new RAIL_SetTxFifo().
193  * The former RAIL_LoadTxData() has been merged into an updated
194  RAIL_WriteTxFifo() API.
195  * The RAIL_Event_t type was renamed \ref RAIL_Events_t now that multiple
196  events can be presented to the application simultaneously in RAIL's
197  generic events callback, rather than relying on the internal arbitrary
198  order that RAIL imposed previously, which may not match the application's
199  or protocol's expectations.
200  * Event/callback configuration is now managed entirely by
201  RAIL_ConfigEvents(), eliminating the old RAIL_ConfigTx() and
202  RAIL_SetCallbackConfig() APIs.
203  * Channel numbers were increased in range from 8 bits to 16 bits,
204  and channel configurations made much more flexible allowing for
205  multi-band radio configurations and an easy way to specify TX power
206  limits on certain channels that otherwise share a common configuration.
207  See new APIs RAIL_GetActiveChannelConfig() and
208  RAIL_GetActiveChannelConfigEntry() along with changes to
209  RAIL_ConfigChannels().
210  * Power levels were given a formal \ref RAIL_TxPowerLevel_t type, and
211  RAIL_SetTxPower() now returns a \ref RAIL_Status_t not the power level
212  actually achieved -- must now use RAIL_GetTxPower() to get that.
213  * PA power curves and the conversion between RAIL power level and dBm
214  have been fully abstracted to live outside of RAIL, allowing customers
215  complete flexibility to characterize their designs and PA selection.
216  See new APIs RAIL_ConfigTxPower(), RAIL_ConvertDbmToRaw(),
217  RAIL_ConvertRawToDbm(), RAIL_GetTxPowerConfig(), RAIL_GetTxPowerDbm(),
218  RAIL_SetTxPowerDbm(), and RAIL_EnablePaCal().
219  * Protocols using Frame Types to determine packet length are better
220  integrated into the Studio radio configurator and channel configuration.
221  Gone are APIs like RAIL_ConfigFrameTypeLength() and
222  RAIL_EnableAddressFilterByFrameType() whose functionality is now
223  subsumed within channel configurations.
224  * APIs to support RAIL timebase synchronization across sleep/wake were
225  added: RAIL_ConfigSleep(), RAIL_Sleep(), and RAIL_Wake().
226  * State transitions for success and error are now encapsulated in a
227  new \ref RAIL_StateTransitions_t structure. This affected the
228  RAIL_SetRxTransitions() and RAIL_SetTxTransitions() routines.
229  * The former RAIL_EVENT_TX_BUFFER_UNDERFLOW has been broken out into
230  non-ACK (\ref RAIL_EVENT_TX_UNDERFLOW) and ACK (\ref
231  RAIL_EVENT_TXACK_UNDERFLOW) flavors like the other
232  transmit completion events.
233  * Diagnostic toning APIs were merged into the streaming APIs as the
234  \ref RAIL_StreamMode_t::RAIL_STREAM_CARRIER_WAVE.
235  * RAIL_ConfigCal() was revamped to use \ref RAIL_CalMask_t.
236  * RAIL_Calibrate()'s bool calSave argument was removed, being
237  redundant with a NULL vs. non-NULL calValues argument.
238  * Support for Packet Traffic Arbitration (PTA) or Coexistence was
239  added in the form of an API to prevent transmits from going on the air:
240  RAIL_EnableTxHoldOff(), along with RAIL_IsTxHoldOffEnabled() to check
241  the current state.
242  * The former RAIL_AutoAckDisable() was merged into RAIL_ConfigAutoAck()
243  by means of its \ref RAIL_AutoAckConfig_t::enable parameter field.
244  * The former RAIL_EnableRxFifoThreshold() was replaced by
245  using RAIL_ConfigEvents() to enable/disable the
246  \ref RAIL_EVENT_RX_FIFO_ALMOST_FULL event.
247  * The former blocking RAIL_PollAverageRssi() was eliminated; use the
248  non-blocking RAIL_StartAverageRssi() instead.
249  * Several APIs were renamed for consistency and updated accordingly:
250  * former RAIL_GetTxPacketInfo() is now RAIL_GetTxPacketDetails();
251  * former RAIL_UseTxBufferForAutoAck() is now RAIL_UseTxFifoForAutoAck();
252  * former RAIL_LoadAutoAckBuffer() is now RAIL_WriteAutoAckFifo().
253  * More API functions were updated to take a \ref RAIL_Handle_t argument:
254  RAIL_SetFreqOffset(), RAIL_GetRxFreqOffset(), and the
255  \ref IEEE802_15_4 "RAIL_IEEE802154_ protocol" APIs.
256  * RAIL_IEEE802154_GetAddress() was updated to return \ref RAIL_Status_t
257  rather than \ref RAIL_IEEE802154_Address_t which is now passed as an
258  output argument.
259  * RAIL_EnableDirectMode() and RAIL_SetTune() now return \ref RAIL_Status_t
260  rather than void.
261  * A multiprotocol scheduler API was added to get its status:
262  RAIL_GetSchedulerStatus().
263  * The RAILCb_AssertFailed() callback returns with a \ref RAIL_Handle_t
264  parameter that might be NULL since assert conditions might occur during
265  initialization prior to handle assignment or during multiprotocol
266  switching.
267  * Many APIs with pointer arguments were clarified to use const pointers
268  where the API doesn't modify the data pointed to.

270 ### 2.0.0
271  * All API names were updated to follow a strictly VerbNoun convention.
272  * RAIL_SetAbortScheduledTxDuringRx() is no longer a separate API, rather it
273  was added to \ref RAIL_ScheduleTxConfig_t. Set the desired TX during RX
274  behavior when that struct is passed to RAIL_StartScheduledTx().
275  * Add support for building single-prototocol, multi-protocol, and core RAIL
276  libraries.
277  * Changed \ref RAIL_RadioState_t to a bitmask. Supported
278  \ref RAIL_RadioState_t bits are: \ref RAIL_RF_STATE_ACTIVE,
279  \ref RAIL_RF_STATE_RX, and \ref RAIL_RF_STATE_TX; combinations of
280  these bits yield the additional states \ref RAIL_RF_STATE_INACTIVE,
281  \ref RAIL_RF_STATE_IDLE, \ref RAIL_RF_STATE_RX_ACTIVE, and
282  \ref RAIL_RF_STATE_TX_ACTIVE.
283  * Added \ref RAIL_RX_OPTION_TRACK_ABORTED_FRAMES to configure seeing full
284  packets via Packet Trace, even after they have been aborted.
285  * Added RAIL 15.4 PTI Rx error code UNDESIRED_ACK to indicate an ACK filtered
286  because it did not have the desired sequence number matching that in the
287  recently-transmitted ACK-requesting frame.
288  * Updated return type from uint8_t to \ref RAIL_Status_t for many functions.
289  * Removed deprecated bit error-rate APIs RAIL_BerConfigSet(),
290  RAIL_BerRxStart(), RAIL_BerRxStop(), RAIL_BerStatusGet(), along
291  with deprecated types RAIL_BerConfig_t and RAIL_BerStatus_t.
292  * TX related events are now broken out to specify whether the event
293  is relevant to an ACK packet or user-initiated TX. Specifically those
294  are \ref RAIL_EVENT_TX_ABORTED and \ref RAIL_EVENT_TXACK_ABORTED,
295  \ref RAIL_EVENT_TX_BLOCKED and \ref RAIL_EVENT_TXACK_BLOCKED, and
296  \ref RAIL_EVENT_TX_PACKET_SENT and \ref RAIL_EVENT_TXACK_PACKET_SENT.
297  One of each type (i.e. one ACK, one non-ACK) can occur before being
298  handled.
299  * RAIL_StartTx() no longer takes a pre-TX operation function pointer to do
300  scheduled or CCA transmits. Instead, there are now separate API's for each
301  type of transmit: RAIL_StartTx(), RAIL_StartScheduledTx(),
302  RAIL_StartCcaLbtTx(), RAIL_StartCcaCsmaTx().
303  * \ref RAIL_TxOptions_t has been converted to a 32-bit bitmask, so that RX
304  and TX options are both represented as bitmasks. A single TX option mask
305  is passed into each TX call.
306  * RAIL_ConfigEvents() now also takes two bitmasks, one to indicate which
307  events should be affected, and the second indicating which of those
308  events should be enabled.
309  * Similarly, RAIL_ConfigRxOptions() takes two bitmasks, one to indicate
310  which options to update, and the second indicating the desired value of
311  those options, which affect all future receives.
312  * Options for removing RX appended info and ignoring CRC are now controlled
313  exclusively by RAIL_ConfigRxOptions().

322 ### 1.6.0
323  * Fixed an issue where RAIL_RfStateGet() would mistakenly return
324  RAIL_RF_STATE_IDLE instead of RAIL_RF_STATE_TX when a
325  Transmit operation's LBT is active.
326  * Added an API (RAIL_SetAbortScheduledTxDuringRx) that sets the behavior
327  when a scheduled TX fires during an RX or ACK. If the API is passed false,
328  TX's will be delayed until the RX (or ACK to the RX) is complete. That
329  behavior is the default and what always happened previously. Passing the
330  function true however will abort scheduled TX's that fire during the RX
331  (or ACK to the RX) and fire the RAILCb_TxRadioStatus callback.
332  * Added new RAILCb_AssertFailed callback to give the flexible handling of
333  asserts within RAIL. This includes defines of assert codes and error
334  reasons in rail_assert_error_codes.h. The default implementation hangs in
335  an infinite loop.
336  * Added RAIL_ENUM define to override enum size ambuiguity in the ARM EABI by
337  making them actually a uint8_t. For documentation purposes they are still
338  shown as enums.
339  * Added new TX options: removeCrc, to override whether CRC is sent; and
340  syncWordId, to choose which SYNC word to transmit when multiple SYNC words
341  are configured.
342  * Added new RX option: RAIL_RX_OPTION_ENABLE_DUALSYNC, which allows multiple
343  SYNC words to be searched for on PHYs that support it.
344  * Added RAIL_PeekRxPacket to allow reading part of a packet before it has
345  been fully received.
346  * Changed BER Test functionality in RAILTest such that if a RX overflow is
347  detected (due to the incoming datarate being too high), the test now aborts
348  and displays an appropriate status message.
349  * Fixed an issue where the code could hang in a RX overflow state when
350  receiving incoming data streams.
351  * RAIL_TxToneStart() and RAIL_TxStreamStart() now idle the radio before
352  transmitting.
353  * Fixed an issue where calling RAIL_TimerSet() while in the middle of a
354  stream or tone transmission could lock up the radio.
355  * Added new frequency offset related functions, RAIL_GetRxFreqOffset and
356  RAIL_SetFreqOffset, and the associated RAIL_FrequencyOffset_t type
357  definition.

372 ### 1.5.1
373  * Added official support for the EFR32xG12 parts through a new
374  librail_efr32xg12.a version of the library.
375  * Added RAIL_SetRxOptions API for configuration of receive options, such as
376  RAIL_RX_OPTION_STORE_CRC.
377  * Increase transmit and receive fifo sizes to 512 bytes for all EFR32xG1
378  parts. This impacts both packet and fifo modes.
379  * Added \ref RAIL_RxDataSource_t::RX_DEMOD_DATA in order to receive
380  demodulated data via FIFO method.
381  * Shortened the duration that the transmit buffer lock in RAIL_TxDataLoad()
382  is held, as it was overprotective. RAIL will still report an error if the
383  application attempts to manipulate the transmit buffer while actively
384  transmitting data.
385  * Improved state transition timings for enhanced precision and accuracy. To
386  improve precision we have switched to a more stable receive complete event
387  since our previous trigger could jitter for certain radio configurations. To
388  improve accuracy we measured and removed PHY specific overhead for BLE
389  applications. Note that this measurement was BLE specific and does _not_
390  completely correct accuracy on any arbitrary PHY configuration. For custom
391  PHY configurations you will still have to manually calibrate out any offsets
392  by measurement and any previous measurements are likely different now.
393  * Moved BER test implementation out of the RAIL library and into RAILTest.
394  Deprecated the existing BER test functions in the RAIL API. With BER test
395  code now existing in RAILTest, you must use version 2.2.1 or later of the
396  radio configurator.
397  * Disabled receive buffer overflow and transmit buffer underflow as
398  BUFC_IRQHandler() events. These events now only occur in FRC_IRQHandler().
399  * Sped up RAIL transmit and receive buffer read/write operations.
400  * Fixed a race condition that could cause the receive packet callback to be
401  delayed until the next receive packet event.
