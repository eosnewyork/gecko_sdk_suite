1 EFR32 {#efr32_main}
2 ============
3 
4 While RAIL attempts to be chip-agnostic, certain hardware specifics
5 can't be overlooked. Where possible, missing features will be simulated in
6 software, but performance characteristics may vary. This section covers EFR32-specific
7 content including hardware-specific configurations or calibrations.
8 
9 ## Clocks
10 
11 ### High Frequency Clocks
12 
13 The EFR32 has a configurable clock tree with a number of different prescalars.
14 One thing to keep in mind, however, is that the radio must always run off of an
15 HFXO at 38.4 MHz. This means that you must switch your HFCLK source to the HFXO
16 before calling RAIL_Init() and make sure to leave the CMU->HFPRESC set to 1 to
17 prevent dividing down the radio clock. You are free to scale the peripheral
18 (CMU->HFPERPRESC) or Cortex (CMU->HFCOREPRESC) clocks using their prescalars
19 which are further down the clock tree. Keep in mind that this may introduce
20 extra wait states when interacting with the radio and slow down some operations.
21 
22 ### Low Frequency Clocks
23 
24 RAIL can use a low frequency clock on the EFR32 to synchronize its time base
25 across EM2 sleep. This only happens if you call \ref RAIL_ConfigSleep with \ref
26 RAIL_SleepConfig_t::RAIL_SLEEP_CONFIG_TIMERSYNC_ENABLED. On the EFR32xG1 and
27 EFR32xG12 this uses the RTCC's clock which is the LFE clock with a possible
28 additional prescalar. The slower the clock source here the longer it will
29 take to synchronize since the process requires several ticks of the RTCC clock.
30 On the EFR32xG13 and EFR32xG14 there is a separate timer to help with this
31 synchronization, but either the LFXO or LFRCO are still used. When calling \ref
32 RAIL_ConfigSleep() the code will first try to setup the timer with the LFXO if
33 it's started, then fallback to the LFRCO, and finally assert if no LF clocks are
34 running.
35 
36 ## Packet Buffer Restrictions
37 
38 The RAIL library for the EFR32 optionally allocates an internal buffer to store
39 receive data contiguously. Other than being byte-aligned, the buffer has no
40 further alignment restrictions. The buffer is set to 512 bytes at build time by
41 default, but can be changed by calling \ref RAIL_SetRxFifo, for example in an
42 implementation of \ref RAILCb_SetupRxFifo. The receive FIFO's size must be a
43 power of 2 between 128 and 4096 for hardware compatibility. This choice limits
44 the maximum size of receive packets in packet mode and determines the size of
45 the receive FIFO in FIFO mode. Since each receive packet has 6 bytes of
46 overhead, you can only receive up to one 506 byte packet without overriding the
47 RX FIFO or switching to FIFO mode. In FIFO mode, you must read out packet data
48 as you approach this limit and store it off to construct the full packet later.
49 Note that the 6 byte overhead may increase or decrease in future releases as
50 the functionality is changed though large jumps are not expected in either
51 direction.
52 
53 The transmit FIFO must be set at runtime with \ref RAIL_SetTxFifo(). The
54 transmit FIFO must also be byte-aligned, with no further restrictions. Its size
55 has the same restrictions as the receive FIFO. The size must be a power of 2
56 from 64 to 4096 for hardware compatibility. Note that there is no difference
57 between packet and FIFO mode on the transmit side. A packet may either be
58 loaded all at once or in pieces using the \ref RAIL_WriteTxFifo() function. You
59 may also use the \ref RAIL_SetTxFifoThreshold() function and the \ref
60 RAIL_EVENT_TX_FIFO_ALMOST_EMPTY event to load data as there is space available
61 in the FIFO during transmission.

63 ## Data Reception Sources
64 
65 When receiving data, you can configure hardware to provide data from three
66 different hardware sources.
67 
68 First, you can configure the hardware to provide a packet of information.
69 This configuration uses the built-in demodulator and frame controller.
70 Use \ref RAIL_RxDataSource_t::RX_PACKET_DATA to enable this receive source.
71 
72 Second, you can configure hardware to provide data directly from the
73 demodulator. In this mode, hardware demodulation is used, but the user is
74 responsible for implementing frame controller functionality. In other words,
75 preamble detection, sync word detection, CRC validation, and so on must be
76 performed in software by the application. All data returned is represented as
77 8-bit 2's-compliment values. Use \ref RAIL_RxDataSource_t::RX_DEMOD_DATA
78 to enable the receive source.
79 
80 Third, you can configure hardware to provide data directly from the I and
81 Q ADCs. In this mode, the user is responsible for implementing demodulator and
82 frame controller functionality. The receive signal hardware has a 19-bit
83 dynamic range. The user can select whether to return the upper 16 bits of
84 the 19-bit value (\ref RAIL_RxDataSource_t::RX_IQDATA_FILTMSB) or the lower 16
85 bits (\ref RAIL_RxDataSource_t::RX_IQDATA_FILTLSB). All data returned is
86 represented as 16-bit 2s-compliment values. The I and Q values are put into the
87 buffer in an alternating fashion where the first 16-bit value is the first I
88 ADC value and the second 16-bit value is the first Q ADC value, and so on.

90 ## Interrupt Vectors
91 
92 The RAIL library for EFR32 implements all of the interrupt vectors for radio
93 peripherals, which is required for the RAIL library to function correctly. RAIL
94 does not, however, set the priorities of these interrupt vectors. That must be
95 handled by your application using the CMSIS NVIC_SetPriority() API or direct
96 manipulation of the NVIC. Below is the full list of interrupts used by the
97 radio. You __must__ run them all at the same priority which is what all Silicon
98 Labs RAIL applications do by default. You are free to choose what that priority
99 is based on the requirements of your application (e.g., putting them below
100 FreeRTOS OS atomic for access to OS functions in RAIL callbacks). Keep in mind that
101 putting the radio interrupts at too low a priority can cause missed packets and
102 other radio performance issues. A restriction on EFR32xG12 and newer is that you
103 must not disable radio interrupts for longer than a quarter of the RAIL timebase
104 (2^32/4 microseconds or 18 minutes) to ensure proper timekeeping. See
105 \ref EFR32xG1x_Interrupts or follow the links below for more information.

122 ## EFR32 Hardware Initialization
123 
124 EFR32 includes EMLIB and EMDRV to create a basic HAL layer.
125 A lot of this initialization code is completely up to you, but there are a
126 couple of requirements when building a RAIL app. Specifically, the radio
127 will only work if you are running off a high-precision crystal oscillator. Since
128 some APIs will assume this is running, make sure to initialize and
129 switch to the crystal before calling any radio APIs.
130 
131 For the Wireless Starter Kit (WSTK), you can use the crystal configuration
132 in the HAL configuration header file
133 for your specific kit. Example code for this is shown below. If you have a
134 custom hardware layout, you may want to create your own HFXOInit structure to
135 account for things such as your specific CTUNE value.
136 
137 ~~~~~~~~~~{.c}
138 #include "bsp.h" // Contains WSTK versions of the HFXO init structure
139 
140 void efrStartup(void)
141 {
142  CMU_HFXOInit_TypeDef hfxoInit = CMU_HFXOINIT_WSTK_DEFAULT;
143 
144  // Initialize the HFXO using the settings from the WSTK bspconfig.h
145  // Note: This configures things like the capacitive tuning CTUNE variable
146  // which can vary based on your hardware design.
147  CMU_HFXOInit(&hfxoInit);
148 
149  // Switch HFCLK to HFXO and disable HFRCO
150  CMU_ClockSelectSet(cmuClock_HF, cmuSelect_HFXO);
151  CMU_OscillatorEnable(cmuOsc_HFRCO, false, false);
152 }
153 ~~~~~~~~~~

155 ## Radio-specific Initialization
156 
157 Currently, \ref RAIL_ConfigTxPower() and \ref RAIL_ConfigPti() initialization
158 functions depend on the board configuration and must be manually called
159 at startup. You __MUST__ initialize the power amplifier (PA) to transmit.
160 You may initialize the packet trace interface (PTI) for use in debugging.
161 
162 
163 ### Power Amplifier (PA) Initialization
164 
165 [High Power 2.4GHz PA](\ref RAIL_TX_POWER_MODE_2P4_HP),
166 [Low Power 2.4GHz PA](\ref RAIL_TX_POWER_MODE_2P4_LP), and
167 [Sub GHz PA](\ref RAIL_TX_POWER_MODE_SUBGIG) PAs are available
168 for the EFR32xG1 family of chips. The
169 specific set of PAs you have available and the supported power levels for those
170 PAs are determined by your part number. See the appropriate data sheet for more details.
171 
172 Each PA supports a raw power level value of type \ref RAIL_TxPowerLevel_t where
173 a numerically smaller number causes less power to be output from the chip than a
174 higher number. Keep in mind that these values may be capped at the upper and
175 lower ends and do not necessarily map linearly to output power in dBm. To map
176 these values to and from dBm values, the RAIL library uses the \ref
177 RAIL_ConvertRawToDbm() and \ref RAIL_ConvertDbmToRaw() APIs. By default, these
178 use whatever piecewise linear power curves were passed into
179 RAIL_InitTxPowerCurves() to do the conversion. On Silicon Labs boards, these
180 curves are included with the board header files. For custom boards, this must be
181 measured using test equipment to take into account the differences in the RF
182 path that may exist. In addition, for even more custom use cases, the conversion
183 functions can be strongly defined in the customer application and provide
184 whatever type of mapping is found to be most effective. Below is an example of
185 using the standard APIs to initialize all PAs on a dual band chip and switch to
186 the 2.4 GHz high-power PA.
187 
188 ~~~~~~~~~~{.c}
189 #include "rail.h"
190 #include "rail_chip_specific.h"
191 #include "plugin/pa-conversion/pa_conversion_efr32.h"
192 
193 RAIL_DECLARE_TX_POWER_VBAT_CURVES(piecewiseSegments, curvesSg, curves24Hp, curves24Lp);
194 
195 // Must be called with a valid RAIL_Handle_t returned by RAIL_Init()
196 void initPa(RAIL_Handle_t myRailHandle)
197 {
198  // Initialize the RAIL Tx power curves for all PAs on this chip
199  RAIL_TxPowerCurvesConfig_t txPowerCurvesConfig = {
200  curves24Hp,
201  curvesSg,
202  curves24Lp,
203  piecewiseSegments
204  };
205  if (RAIL_InitTxPowerCurves(&txPowerCurvesConfig) != RAIL_STATUS_NO_ERROR) {
206  // Could not initialize transmit power curves so something is configured
207  // wrong. Please fix and rebuild.
208  while(1);
209  }
210 
211  // Switch to the 2.4GHz HP PA powered off the 1.8V DCDC connection
212  RAIL_TxPowerConfig_t railTxPowerConfig = {
213  RAIL_TX_POWER_MODE_2P4_HP, // 2.4GHz HP Power Amplifier mode
214  1800, // 1.8V vPA voltage for DCDC connection
215  10 // Desired ramp time in us
216  };
217  if (RAIL_ConfigTxPower(myRailHandle, &railTxPowerConfig)
218  != RAIL_STATUS_NO_ERROR) {
219  // Error: The requested PA could not be selected. Fix your configuration
220  // and try again.
221  while(1);
222  }
223 
224  // Set the output power to the maximum supported by this chip
225  RAIL_SetTxPower(myRailHandle, 255);
226 }
227 ~~~~~~~~~~

229 ### Packet Trace Interface (PTI) Initialization
230 
231 Packet trace on the EFR32 provides a mechanism for viewing transmitted and
232 received radio packets for network sniffing or debugging. It can also be
233 captured by a WSTK and sent to Simplicity Studio for viewing data in its
234 Network Analyzer tool.
235 
236 To use this functionality, configure the pins to
237 use for this output and, optionally, how to format data. Note that
238 the WSTK _requires_ the following output format:
239 
240  Mode: 8 bit UART mode
241  Baudrate: 1.6Mbps
242  Framing Signal: Enabled
243 
244 Introducing changes is currently unsupported by the WSTK.
245 Other output formats are currently unsupported.
246 
247 To choose the pins, look in the applicable data sheet and
248 select a valid route location for the FRC_DOUT and FRC_DFRAME signals. These map
249 to PTI.DATA and PTI.FRAME on the WSTK respectively. Once you've found pins that
250 work for your hardware, configure the RouteLocation, port, and pin
251 fields in the PTI initialization structure. When using the WSTK for example,
252 initialize as follows:
253 
254 ~~~~~~~~~~{.c}
255 #include "rail.h"
256 #include "rail_chip_specific.h"
257 
258 void initPti(void)
259 {
260  RAIL_PtiConfig_t ptiConfig = {
261  RAIL_PTI_MODE_UART, // Only supported output mode for the WSTK
262  1600000, // Choose 1.6 Mbps for the WSTK
263  6, // WSTK uses location 6 for DOUT
264  gpioPortB, // FRC_DOUT#6 is PB12
265  12, // FRC_DOUT#6 is PB12
266  6, // UNUSED IN UART MODE
267  gpioPortB, // UNUSED IN UART MODE
268  11, // UNUSED IN UART MODE
269  6, // WSTK uses location 6 for DFRAME
270  gpioPortB, // FRC_DOUT#6 is PB13
271  13, // FRC_DOUT#6 is PB13
272  };
273 
274  // Initialize the Packet Trace Interface (PTI) for the EFR32
275  RAIL_ConfigPti(RAIL_EFR32_HANDLE, &ptiConfig);
276 
277  // Enable Packet Trace (PTI)
278  RAIL_EnablePti(RAIL_EFR32_HANDLE, true);
279 }
280 ~~~~~~~~~~

282 ### Other Radio GPIO Functions
283 
284 Various useful signals related to the radio can be output on a
285 GPIO using the Peripheral Reflex System (PRS). PRS is an advanced system
286 where you can route signals to channels and then output those channels to a
287 number of configurable locations. For more information,
288 see the PRS chapter in the reference manual.
289 
290 A list of some of the most interesting PRS signals related to the radio is shown
291 below along with how to enable them in the PRS. The definition for these signals
292 for a given chip can be found in the release at
293 `platform/Device/SiliconLabs/<chipFamily>/Include/<chipFamily>_prs_signals.h`
294 where `<chipFamily>` would be the beginning of the part number (EFR32MG12P for
295 example).
296 
297 | Signal | Summary |
298 |:--------------------|:----------------------|
299 | PRS_RAC_ACTIVE | Radio enabled |
300 | PRS_RAC_TX | Transmit mode enabled |
301 | PRS_RAC_RX | Receive mode enabled |
302 | PRS_RAC_LNAEN | LNA enabled for Rx |
303 | PRS_RAC_PAEN | PA enabled for Tx |
304 | PRS_MODEM_FRAMEDET | Frame detected |
305 | PRS_MODEM_PREDET | Preamble detected |
306 | PRS_MODEM_TIMDET | Timing detected |
307 | PRS_MODEM_FRAMESENT | Frame sent |
308 | PRS_MODEM_SYNCSENT | Syncword sent |
309 | PRS_MODEM_PRESENT | Preamble sent |
310 
311 The example below shows how to configure a PRS channel to output RAC_RX on
312 a GPIO. In this example, the WSTK is used with the BRD4153A radio
313 board. The code below will put RAC_RX on PRS Channel 0 and output PRS
314 Channel 0 on pin PC10 which is wired to WSTK_P12 and EXP_HEADER15 on the WSTK.
315 
316 ~~~~~~~~~{.c}
317 #include "em_cmu.h"
318 #include "em_prs.h"
319 #include "em_gpio.h"
320 #include "em_device.h"
321 
322 void enableDebugGpios(void)
323 {
324  // Turn on the PRS and GPIO clocks to access their registers.
325  CMU_ClockEnable(cmuClock_PRS, true);
326  CMU_ClockEnable(cmuClock_GPIO, true);
327 
328  // Configure PC10 as an output.
329  GPIO_PinModeSet(gpioPortC, 10, gpioModePushPull, 0);
330 
331  // Configure PRS Channel 0 to output RAC_RX.
332  PRS_SourceSignalSet(0,
333  ((PRS_RAC_RX & _PRS_CH_CTRL_SOURCESEL_MASK)
334  >> _PRS_CH_CTRL_SOURCESEL_SHIFT),
335  ((PRS_RAC_RX & _PRS_CH_CTRL_SIGSEL_MASK)
336  >> _PRS_CH_CTRL_SIGSEL_SHIFT),
337  prsEdgeOff);
338 
339  // Configure PRS Channel 0 to use output location 12 (PC10 - see data sheet).
340  PRS->ROUTELOC0 &= ~_PRS_ROUTELOC0_CH0LOC_MASK;
341  PRS->ROUTELOC0 |= PRS_ROUTELOC0_CH0LOC_LOC12;
342 
343  // Enable PRS Channel 0.
344  PRS->ROUTEPEN |= PRS_ROUTEPEN_CH0PEN;
345 }
346 ~~~~~~~~~

366 ## Peripherals Consumed by RAIL
367 
368 Certain functionality in RAIL requires some of the chip's peripherals. The
369 specifics of these requirements are enumerated below.
370 
371 ### RAIL Timer Synchronization
372 
373 If you call \ref RAIL_ConfigSleep with \ref
374 RAIL_SleepConfig_t::RAIL_SLEEP_CONFIG_TIMERSYNC_ENABLED to keep the clock
375 synchronized across sleep RAIL must use the PRS and RTCC.
376  * On all platforms PRS channel 7 is used to perform the synchronization.
377  This channel is only initialized one time when calling the \ref
378  RAIL_ConfigSleep function so you must not change it after that or
379  the synchronization may fail.
380  * On the EFR32xG1 and EFR32xG12 platforms the timer synchronization also uses
381  RTCC channel 0. This channel should not be used by the application after a
382  call to \ref RAIL_Sleep and before the corresponding \ref RAIL_Wake call.
383  The application is also responsible for enabling this top level interrupt if
384  they want it to serve as a wake source and implementing an interrupt handler
385  that clears the RTCC_IFC_CC0 flag in the RTCC->IF register any time it is
386  pended to prevent the chip from getting stuck in the handler. RAIL does not
387  need to be tied into the interrupt handler in any other way.
388 
389 ## Entropy Generation
390 
391 EFR32 supports true entropy collection using the radio. It is able to
392 collect 1 bit per radio clock cycle while in receive mode. This means that
393 the receiver must be enabled when collecting entropy. If you attempt to transmit
394 or otherwise delay entry into receive, data collection will take longer.
395 You can still receive packets while collecting entropy. This
396 means that requesting random numbers longer than the length of your preamble and
397 sync word may trigger a packet reception. This may be fixed in a future revision
398 but for now it ensures a proper packet reception during a random data
399 collection.
400 
401 Due to the nature of random data collection, either the number of bytes or zero
402 is returned after the full amount requested.
403 Zero is returned if the radio is uninitialized and cannot be enabled for
404 collection.
405 
406 ## RAIL Timebase
407 
408 EFR32 uses a dedicated radio timer to create the RAIL timebase and perform
409 scheduled transmit and receive operations. This timer ticks at roughly every 2 us
410 on the EFR32xG1 platform and 0.5 us on the EFR32xG12 and newer. The exact tick
411 value depends on the high-frequency crystal in use so there may be a small
412 rounding error. All internal APIs account for this error, so over long periods of
413 time it is generally unnoticeable. Note that RAIL uses the hardware tick rate to
414 create a 1 us time base that is exposed to the user. This is all that should
415 matter for most users and the information here is included to explain
416 the underlying implementation.
417 
418 ## Radio Calibration
419 
420 EFR32 supports image rejection (IR)
421 calibration and VCO temperature calibration. You may choose to enable one or
422 both depending on your use case. Using
423 both is recommended for optimal performance of the radio.
424 
425 * Image Rejection Calibration (IRCAL)
426  * This calibration should be run each time your PHY configuration changes. It
427  is based on the modulation scheme, frequency band, and other radio settings.
428  RAIL will request this by calling \ref RAIL_Config_t::eventsCallback with the \ref
429  RAIL_EVENT_CAL_NEEDED bit set and the \ref RAIL_CAL_ONETIME_IRCAL bit set
430  in \ref RAIL_GetPendingCal().
431  * Using a proper value for this will improve sensitivity by several dBm, so
432  it's highly recommended.
433  * It can take on the order of 700 ms to complete this calibration.
434  You may want to save off a known good value for this calibration and load
435  it each time you switch PHYs to save time.
436  * This calibration should be initialized before calling
437  \ref RAIL_ConfigChannels() to ensure that it is properly configured by the
438  time the first channel is set up. The initialization involves enabling the
439  algorithm and passing in parameters to configure the algorithm for the
440  specific PHY generated by the EFR32 Radio Configurator.
441  * You cannot use the radio while this calibration is being performed or you
442  may generate an incorrect calibration. Application code should ensure that
443  the radio remains in the idle state during this calibration.
444  * This calibration is only meaningful for Zigbee, BLE, and sub GHz radio
445  configurations. For all others you should still enable the calibration but
446  the algorithm will apply a safe, default, calibration value.
447 * VCO Temperature Calibration (VCO_TEMPCAL)
448  * When staying in receive for a very long time and the temperature changes
449  causing the radio to drift off-frequency, RAIL will request this calibration
450  via \ref RAIL_Config_t::eventsCallback with
451  the \ref RAIL_EVENT_CAL_NEEDED bit set and the \ref RAIL_CAL_TEMP_VCO bit
452  set in RAIL_CalPendingGet().
453  * This calibration is automatically run every time receive is entered. If the
454  application, by its nature, frequently re-enters receive mode, this
455  calibration may not need to be enabled.
456  * On EFR, the application will get this event when the absolute temperature
457  crosses 0C degrees as well as when the temperature delta from the last
458  calibration increases or decreases by 70C.
459  * It is always recommended to enable and handle this calibration since it
460  doesn't add much overhead and is much safer.
461 
462 ## State Transition Timing
463 
464 EFR32 allows automatic transitions from receive to transmit to
465 precisely time transmitted packets after packet reception. During this process,
466 internal calculations are performed which require the received packet's
467 duration, from sync word to end of CRC, to be less than 32 ms. If received
468 packets in a given protocol have an on-air time greater than 32 ms, the
469 automatic transition from receive to transmit may not be used with that
470 protocol because of incorrect timing of that transition.
471 
472 ## Radio State Verification
473 
474 RAIL includes a radio state verification feature capable of verifying radio
475 register contents. There are multiple ways of configuring and running the
476 verification process. All radio state verification should occur when the radio
477 is idle. If not idle, the number of radio registers different
478 from their reference values varies at any given time.
479 
480 ### Default vs. Custom Radio Config
481 
482 When verification occurs, a reference value is compared with a radio register's
483 contents. The reference value is provided through one of two ways.
484 
485 The radio configuration (originally passed into \ref RAIL_ConfigChannels())
486 is used as a reference with which to compare radio register values. When the
487 default radio configuration is used for verification, only those addresses flagged
488 as being verifiable by the radio configurator will be checked. Bit 27 of an
489 encoded address indicates whether that address is verifiable or not.
490 For example, encoded address 0x08020004 is flagged as being verifiable and
491 will be verified when \ref RAIL_Verify() is called, but the encoded
492 address 0x00020008 will not be verified.
493 
494 A custom radio configuration can be provided to the verification API (passed
495 into \ref RAIL_ConfigVerification()). When a custom radio configuration is provided,
496 all addresses in the provided radio configuration will be checked, regardless of
497 whether or not their encoded addresses are flagged as being verifiable.
498 
499 ### Approval Callback
500 
501 When verification occurs, no register value differences are allowed unless
502 an application-level approval callback is defined.
503 
504 Without an approval callback, any difference will be flagged as data
505 corruption, with \ref RAIL_Verify() returning a
506 status of \ref RAIL_STATUS_INVALID_STATE.
507 
508 With an approval callback, the application can scrutinize all
509 differences and deem them acceptable or not. By providing an approval callback,
510 individual register fields, instead of the entire register contents, can be
511 compared against reference values at the application level.
512 
513 ### Restarting
514 
515 Verification can be restarted if necessary.
516 
517 When running verification from the beginning to completion (by
518 specifying a sufficiently long test duration or by specifying a duration of
519 \ref RAIL_VERIFY_DURATION_MAX), the verification process is indifferent to the
520 restart parameter.
521 
522 When running verification for a duration that does not permit
523 verification to run to completion, the \ref RAIL_Verify() function will return
524 with a status of \ref RAIL_STATUS_SUSPENDED. This indicates test success for
525 those values already verified, but sufficient time was not permitted to verify
526 all values. In this scenario, the initial run of \ref RAIL_Verify() is
527 indifferent to the value of the restart input, but subsequent runs of
528 \ref RAIL_Verify() should set the restart input to false so that verification
529 will resume where it left off on the previous call. To
530 record where a previous verification left off, the \ref RAIL_VerifyConfig_t
531 structure must be declared by the application and provided to RAIL. These
532 structure contents should only be altered by RAIL.
533 
534 ### Configuration and Running Options
535 
536 With the ability to specify default radio configuration vs. custom radio
537 configuration, approval callback vs. no approval callback, and run to
538 completion vs. run for a specific time, there are multiple ways in which
539 to run RAIL's radio state verification feature. The example below shows
540 running verification in each of these modes.
541 
542 ~~~~~~~~~~{.c}
543 #include "rail.h"
544 #include "rail_types.h"
545 #include "rail_config.h"
546 
547 RAIL_VerifyConfig_t configVerify;
548 
549 const uint32_t customRadioConfig[] = {
550  0x00010048UL, 0x00000000UL,
551  0x000400A0UL, 0x00004CFFUL,
552  /* 00A4 */ 0x00000000UL,
553  /* 00A8 */ 0x00004DFFUL,
554  /* 00AC */ 0x00000000UL,
555  0x00012000UL, 0x00000744UL,
556  ...
557  0xFFFFFFFFUL,
558 };
559 
560 bool RAILCb_VerificationApproval(uint32_t address,
561  uint32_t expectedValue,
562  uint32_t actualValue)
563 {
564  responsePrint("verifyRadioCb",
565  "address:0x%08x,expectedValue:0x%08x,actualValue:0x%08x",
566  address, expectedValue, actualValue);
567  return true OR false; // true = change approved; false = change unapproved
568 }

570 main ()
571 {
572  RAIL_Status_t result;
573  ...
574  // Initialize the radio.
575  railHandle = RAIL_Init(..., ...);
576  ...
577  // Associate a radio config with this RAIL handle.
578  RAIL_ConfigChannels(railHandle, channelConfigs[0], ...);
579  ...
580  // Idle the radio before verifying registers.
581  RAIL_Idle(railHandle, RAIL_IDLE, true);
582 
583  uint8_t verifyConfigOption = 1; // 1,2,3,4
584  uint8_t verifyRunOption = 1; // 1,2
585 
586  // Configure the radio state verification in one of 4 ways:
587  if (1 == verifyConfigOption) {
588  // Config Option 1: Use default radio config with no approval callback.
589  RAIL_ConfigVerification(railHandle,
590  &configVerify,
591  NULL, // use channelConfigs[0] for verification
592  NULL);
593  } else if (2 == verifyConfigOption) {
594  // Config Option 2: Use default radio config with approval callback.
595  RAIL_ConfigVerification(railHandle,
596  &configVerify,
597  NULL, // use channelConfigs[0] for verification
598  RAILCb_VerificationApproval);
599  } else if (3 == verifyConfigOption) {
600  // Config Option 3: Use custom radio config with no approval callback.
601  RAIL_ConfigVerification(railHandle,
602  &configVerify,
603  &customRadioConfig[0],
604  NULL);
605  } else {
606  // Config Option 4: Use custom radio config with approval callback.
607  RAIL_ConfigVerification(railHandle,
608  &configVerify,
609  &customRadioConfig[0],
610  RAILCb_VerificationApproval);
611  }
612 
613  // Run the radio state verification in one of 2 ways:
614  if (1 == verifyRunOption) {
615  // Run Option 1: Run verification once to completion.
616  result = RAIL_Verify(railHandle,
617  &configVerify,
618  RAIL_VERIFY_DURATION_MAX,
619  true); // run from the beginning of verification
620  } else {
621  // Run Option 2: Run verification for a specific time before returning.
622  result = RAIL_Verify(railHandle,
623  &configVerify,
624  100, // run for 100 microseconds before returning
625  true); // run from the beginning of verification
626 
627  while (RAIL_STATUS_SUSPENDED == result) {
628  responsePrint("verifyRadio",
629  "verification suspended - run again to continue");
630  result = RAIL_Verify(railHandle,
631  &configVerify,
632  100, // run for 100us before returning
633  false); // run from where verification left off
634  }
635  }
636 
637  switch (result) {
638  case RAIL_STATUS_NO_ERROR:
639  {
640  responsePrint("verifyRadio", "success, done");
641  break;
642  }
643  case RAIL_STATUS_INVALID_PARAMETER:
644  {
645  responsePrint("verifyRadio", "invalid input parameter");
646  break;
647  }
648  case RAIL_STATUS_INVALID_STATE:
649  default:
650  {
651  responsePrint("verifyRadio", "data corruption");
652  }
653  }
654  ...
655 }
